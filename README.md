# Bitcoin Transaction Experiment: Legacy (P2PKH) and SegWit (P2SH-P2WPKH)

## Overview

This project explores how Bitcoin transactions work by creating and validating transactions using two different address formats:

- **Legacy Transactions (P2PKH)**
- **SegWit Transactions (P2SH-P2WPKH)**

The experiment runs on the **Bitcoin Regtest network**, which is a private local Bitcoin network designed for testing and development. Regtest allows us to mine blocks instantly and test transactions without using real Bitcoin.

Two Python scripts are used in this project:

- `p2pkh_transactions.py` → creates and validates **Legacy P2PKH transactions**
- `segwit_transactions.py` → creates and validates **SegWit P2SH-P2WPKH transactions**

Both scripts perform the following steps:

1. Connect to a Bitcoin Core node running in **regtest mode**
2. Generate Bitcoin addresses
3. Fund an address using mined coins
4. Create raw transactions
5. Sign and broadcast transactions
6. Decode transactions and inspect their scripts

# Bitcoin Core Setup (Regtest Mode)

## Step 1 — Install Bitcoin Core

Download Bitcoin Core from:

https://bitcoincore.org/en/download/

Verify installation:

```
bitcoind --version
bitcoin-cli --version
```

---

## Step 2 — Create bitcoin.conf

Create the Bitcoin configuration file.

Location (Linux / Mac):

```
~/.bitcoin/bitcoin.conf
```

Add the following configuration:

```
regtest=1
server=1
rpcuser=user
rpcpassword=password
fallbackfee=0.00001
txconfirmtarget=1
```

---

## Step 3 — Start the Bitcoin Node

Start Bitcoin Core in **regtest mode**:

```
bitcoind -regtest -daemon
```

Check if the node is running:

```
bitcoin-cli -regtest getblockchaininfo
```

# Script Validation using btcdeb

Bitcoin scripts were verified using the **Bitcoin Script Debugger (btcdeb)**.

## Install btcdeb

```
git clone https://github.com/bitcoin-core/btcdeb
cd btcdeb
./autogen.sh
./configure
make
sudo make install
```

## Run Script Debugger

Example command:

```
btcdeb '<signature> <publicKey> OP_DUP OP_HASH160 <pubKeyHash> OP_EQUALVERIFY OP_CHECKSIG'
```

btcdeb executes each opcode step by step and shows how the stack changes.

If the final stack value is:

```
01
```

the script execution is successful.

# Transaction Comparison

| Transaction | Size (bytes) | Vsize (vbytes) | Weight (WU) |
|-------------|--------------|---------------|-------------|
| Legacy A → B | 225 | 225 | 900 |
| Legacy B → C | 225 | 225 | 900 |
| SegWit A′ → B′ | 247 | 166 | 661 |
| SegWit B′ → C′ | 247 | 166 | 661 |

SegWit transactions have a **smaller virtual size**, which reduces transaction fees.

---

# Fee Reduction Calculation

Legacy transaction virtual size:

```
225 vbytes
```

SegWit transaction virtual size:

```
166 vbytes
```

Fee savings:

```
Savings = (1 − 166 / 225) × 100
Savings ≈ 26%
```

This shows that SegWit transactions reduce the effective transaction size and therefore lower the transaction fees.

# Conclusion

This experiment demonstrates how Bitcoin transactions are created and validated using both **Legacy P2PKH** and **SegWit P2SH-P2WPKH** address formats.

Legacy transactions store signatures in the `scriptSig`, which increases transaction size and weight. SegWit separates the signature data into the **witness field**, which receives a lower weight when calculating transaction size.

As a result, SegWit transactions have a **smaller virtual size and lower transaction fees**. The scripts were also verified using **btcdeb**, confirming that the unlocking scripts correctly satisfy the locking script conditions.

Overall, the experiment highlights how SegWit improves **transaction efficiency, scalability, and reliability** in the Bitcoin network.
